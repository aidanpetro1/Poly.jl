## Recommended approach: cumulative tag

The cleanest path is a **single cumulative tag** `v0.7.0` covering
everything from `v0.3.1..HEAD`. Rationale:

- Intermediate work was never tagged, so backdating tags requires
  identifying boundary commits — laborious and error-prone.
- Downstream consumers don't have prior pinned versions of any
  intermediate version, so there's no compatibility constraint to
  preserve.
- The CHANGELOG already separates `[Unreleased]` entries by version,
  which serves as the migration record.

Under this approach, **`v0.7.0` is interpreted as "everything since
`v0.3.1`"**, with the CHANGELOG providing per-feature granularity.

matches actual code commits), the intermediate-tags approach (ROADMAP
"Option B") would tag:

- `v0.4.0` at the v0.4 commit
- `v0.4.1` at the v0.4.x commit
- `v0.5.0` at the v0.5 commit
- `v0.5.1` at the v0.5.1 commit
- `v0.6.0` at the v0.6 commit (or `v0.6 AbstractLens` + `v0.6` Cat#

This is more git work and requires identifying the boundary commits
in `git log`. Recommendation: skip — the cumulative approach is
cleaner.
